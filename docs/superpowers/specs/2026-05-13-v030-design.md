# v0.3.0 Design — Notifications, Haptics, First-Run, Insights v2 Entry

**Date:** 2026-05-13
**Status:** Approved (autonomous mode)
**Target tag:** v0.3.0
**Builds on:** v0.2.12

## Why this ship

v0.2.x focused entirely on visual polish. v0.3.0 introduces the first features that change *what the app does*:

- **Notifications** (Sub-project E from the v0.2 decomposition) — payday-style alerts so the user gets a nudge on the 15th and last day of each month.
- **Haptic feedback** — converts the visual press feedback from v0.2.11 into a multi-sensory tactile experience.
- **First-run experience** — onboarding welcome flow + swipe-to-pay coachmark. Both gated on schema flags so existing users don't see them.
- **Insights v2 entry: per-debt payoff calendar** — a long-asked-for view of "when does each debt finish?"

Out of scope for v0.3.0:
- **Sub-project F (multiplatform expansion)** — needs Mac + Xcode for iOS, separate hardware effort.
- **Loading shimmer** — needs async data layer refactor; current SQLDelight loads are fast enough.
- **Semantic color token refactor** — low user-visible payoff.
- Other Insights v2 items (by-category bar chart, snowball ranking, tap-to-drill, time-horizon switcher) — sequenced for v0.3.1+.

## Architecture

### Schema changes (DB version 3 → 4)

New migration `composeApp/src/commonMain/sqldelight/com/snowball/db/3.sqm`:

```sql
ALTER TABLE Settings ADD COLUMN firstLaunchSeen INTEGER NOT NULL DEFAULT 0;
ALTER TABLE Settings ADD COLUMN swipeCoachmarkSeen INTEGER NOT NULL DEFAULT 0;

-- Existing users (single Settings row already exists) skip both first-run flows.
-- Fresh installs (Settings.insertIfMissing creates the row with column defaults) see both.
UPDATE Settings SET firstLaunchSeen = 1, swipeCoachmarkSeen = 1 WHERE id = 1;
```

`Settings.sq` schema becomes 7 columns: id, incomePerCutoff, notificationsEnabled, notificationHour, notificationMinute, firstLaunchSeen, swipeCoachmarkSeen.

Add `Settings.sq` queries: `setFirstLaunchSeen`, `setSwipeCoachmarkSeen`.

### Haptic feedback (expect/actual)

`composeApp/src/commonMain/kotlin/com/snowball/platform/Haptics.kt`:

```kotlin
expect class Haptics {
    fun tick()       // light tap — tab change, button press, undo
    fun thump()      // heavier — mark paid, debt complete
    fun warn()       // sharper double — validation error (optional, future use)
}

@Composable expect fun rememberHaptics(): Haptics
```

`composeApp/src/androidMain/kotlin/com/snowball/platform/Haptics.android.kt`:

Backed by `androidx.compose.ui.hapticfeedback.HapticFeedback` (Compose's built-in haptic API) — `LocalHapticFeedback.current.performHapticFeedback(HapticFeedbackType.LongPress)` for thump, `.TextHandleMove` for tick. No permission required.

Applied at call sites:
- Tab tap → tick
- Mark-paid (swipe completes) → thump
- Undo paid (swipe back) → tick
- Save button → tick
- Settings toggle → tick

### Notifications (Sub-project E)

**Architecture:** Use `androidx.work` `PeriodicWorkRequest` (daily) with a `CoroutineWorker` that fires every 24 hours at the user's preferred time and posts a notification if today is the 15th or the last day of the month. WorkManager handles boot persistence — no `BootReceiver` needed.

**Files:**
- Common: `composeApp/src/commonMain/kotlin/com/snowball/platform/NotificationScheduler.kt` (expect interface)
- Android: `composeApp/src/androidMain/kotlin/com/snowball/platform/NotificationScheduler.android.kt` (actual, wraps WorkManager)
- Android worker: `composeApp/src/androidMain/kotlin/com/snowball/notifications/PaydayWorker.kt`
- Android channel init: `MainActivity.kt` `onCreate` calls `NotificationChannelInit.create(this)`
- Settings UI: `SettingsScreen.kt` adds toggle + time picker rows; `SettingsViewModel.kt` exposes the three notification fields + setters

**expect API:**
```kotlin
expect class NotificationScheduler {
    fun schedule(enabled: Boolean, hour: Int, minute: Int)
    fun cancel()
    suspend fun requestPermissionIfNeeded(): Boolean
}
```

`schedule(true, hour, minute)` enqueues a daily `PeriodicWorkRequest` aligned to the next occurrence of `hour:minute`. `schedule(false, ...)` cancels. The worker logic:

```kotlin
class PaydayWorker(...) : CoroutineWorker(...) {
    override suspend fun doWork(): Result {
        val today = ... // current LocalDate
        val lastDayOfMonth = today.month.length(today.isLeapYear)
        if (today.dayOfMonth == 15 || today.dayOfMonth == lastDayOfMonth) {
            postPaydayNotification()
        }
        return Result.success()
    }
}
```

**Notification content:**
- Title: "Payday — Snowball"
- Body: "Time to settle this cutoff." (later: dynamic with cutoff totals; v0.3.0 ships the static body)
- Tap → opens app to Home

**Permission flow:**
- `<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />` in manifest (API 33+ runtime permission)
- When the user flips the Settings toggle ON on API 33+, request `POST_NOTIFICATIONS` at runtime. If denied, revert the toggle.
- API < 33: no runtime permission needed; toggle just flips state.

**Build dep:**
- `androidx.work:work-runtime-ktx:2.10.0` — added to androidMain dependencies via gradle catalog.

### First-run experiences

#### Onboarding (welcome flow)

Triggered on app start when `firstLaunchSeen == 0`. Renders BEFORE the `Route.Tabs` content. Three slides + a final "Get started" slide that segues into the Settings screen for income entry.

Slides (each is a full-screen Compose composable with snowflake icon + headline + body + page indicator):
1. **"Track every debt."** — explains the concept of cutoff-based payment tracking.
2. **"Swipe to mark paid."** — illustrates the swipe gesture (small animated chevron).
3. **"Forecast your freedom."** — points to Insights.
4. **"Get started."** — CTA button "Set your income" → flips `firstLaunchSeen = 1` → navigates to Settings.

Pager: simple horizontal slide via `AnimatedContent` directional transition (similar to App.kt route transitions). Page-indicator dots animate active.

State management: a new `Route.Onboarding` case, conditionally chosen at App start.

#### Swipe coachmark

Triggered on the Home screen when `swipeCoachmarkSeen == 0` AND there's at least one unpaid payment row. Renders as an overlay near the first unpaid row, showing a translucent hand-pointer-style chevron icon animating leftward (the swipe direction).

Auto-dismissed after:
- 5 seconds elapsed
- Any swipe gesture detected (success or attempt)
- User taps anywhere (one-finger tap)

Flips `swipeCoachmarkSeen = 1` on dismiss.

Implementation: a `SwipeCoachmark` composable rendered as a Box overlay above the first unpaid row in the LazyColumn. Animation: chevron + finger glyph slides 0 → -40dp → 0 repeatedly with a 1s cycle.

### Insights v2 entry: per-debt payoff calendar

New section on InsightsScreen between SnapshotCard and "UPCOMING (NEXT 6 MONTHS)". Heading: `"PAYOFF TIMELINE"`. Content: a vertical list of all active scheduled debts (`CategoryBehavior.SCHEDULED`, not archived) sorted ascending by `projectedEndDate(debt)` (already available in `JourneyCalculator`).

Each row renders:
- Debt category icon + name (left)
- Projected end date formatted "MMM yyyy" (right, with full date in supporting text below)
- Monthly amount (small, muted)

Empty state: when no scheduled debts active, "No active debts — you're free." (italic, mute).

`InsightsViewModel` exposes `val payoffTimeline: List<PayoffRow>` where `PayoffRow(debt: Debt, endDate: LocalDate, monthlyAmount: Double)`.

## Acceptance

- DB migration runs cleanly on upgrade from v0.2.12 schema (3 → 4). Existing user's settings preserved; `firstLaunchSeen` and `swipeCoachmarkSeen` both = 1 for upgraders.
- Settings screen shows a Notifications toggle + time picker rows after Income. Toggling on:
  - Requests `POST_NOTIFICATIONS` permission on API 33+; reverts toggle if denied.
  - Enqueues the daily WorkManager job.
- Toggling off cancels the job.
- Changing the time re-schedules.
- Notification channel exists; tapping notification opens the app.
- Haptic feedback fires on tab tap, mark-paid, undo, save button, notification toggle.
- Fresh-install on emulator shows onboarding flow → tapping CTA navigates to Settings → entering income proceeds to Home.
- Upgrade install (existing data) skips onboarding.
- Fresh-install Home view (after setting income + adding a debt) shows the swipe coachmark on the first unpaid row.
- Coachmark dismisses on swipe, tap, or 5s elapsed; doesn't re-appear next session.
- Insights screen shows PAYOFF TIMELINE section with sorted debts.
- App builds and APK installs cleanly.

## Risks

- **WorkManager scheduling drift.** Periodic jobs can drift by several minutes. For payday notifications a few minutes' drift is fine. Mitigation: accept the drift; document it.
- **Permission UX on API 33+.** If the user denies notification permission, we silently revert the toggle. Could be confusing. Mitigation: small `supportingText` under the toggle when in this state ("Allow notifications in System Settings to enable").
- **Onboarding rendering before DB is ready.** App.kt initial composition needs to read `firstLaunchSeen` synchronously. Mitigation: load Settings on `viewModel.init` before showing Tabs/Onboarding (single SQLite read, fast).
- **Coachmark positioning over a virtualized list.** If the LazyColumn is mid-scroll, the coachmark could float over the wrong row. Mitigation: tie coachmark visibility to "row index 0 is visible" check via `LazyListState`.
- **Migration safety.** The migration runs an UPDATE on the single Settings row; if for some reason that row doesn't exist on an in-progress install, the UPDATE is a no-op (no error) — `insertIfMissing` re-creates it on first repo read with the new defaults. Verified safe.
