# Snowball — Independent UX/UI Inspection (2026-05-12)

## TL;DR
The visual system is confident — Fraunces display + DM Sans body, an ice/champagne palette, and a strong cutoff hero card on Home land well. But first use is sabotaged by three concrete problems: only one debt category exists with no way to add more, a debt added today vanishes from Home with no explanation, and destructive actions (Save validation fail, Delete) give the user zero feedback. Together those make the empty-state journey feel half-built rather than polished.

## Strengths
- **Home cutoff card is the clear hero.** Display-large 96sp Fraunces W300 numeral plus italic-smaller "₱" glyph (PesoText, `style.fontSize * 0.55f`) reads as serious typography rather than off-the-shelf material. The 28dp corner radius, linear gradient `CardElev → NightElev`, and 1dp `LineStrong` border (`CutoffCard.kt:36-44`) feel intentional.
- **Strong color discipline.** The palette is named, narrow, and consistent (`Color.kt`). Ice (`#9FCEE3`) is reserved for the primary action FAB and LEFT-OVER value, Ember for destructive, Champagne for undo — the symbolic mapping holds across screens.
- **Two-tier type pairing.** Fraunces for amounts and section headers, DM Sans for everything else, with a real type ramp (`Type.kt`) from 96sp display down to 10sp label with 1.5sp tracking. The display-vs-body contrast lands on Home and Debts.
- **Bottom-nav touch targets pass.** Each tab is `[0|360|720,2186][360|720|1080,2312]` = 137×48 dp at density 420 — meets the 48dp minimum.
- **The marked-paid mechanism actually persists.** Edit-form for Sloan reveals "Payments already made = 1" after a single Home tap, so the data path works end-to-end. (See `snowball-form-edit-sloan.xml`.)
- **PesoText number formatting is correct.** Thousand-separator and two-decimal rendering (`PesoText.kt:41-50`) handle ₱9,999,999.99 cleanly.
- **Swipe semantics are gated by paid state.** `enableDismissFromEndToStart = !row.isPaidThisCycle, enableDismissFromStartToEnd = row.isPaidThisCycle` (`HomeScreen.kt:122-123`) prevents nonsensical actions. Good.

## Issues by severity

### Blocking (will frustrate first use)

1. **A debt created today is silently filtered out of Home.**
   `CutoffCalculator.kt:30` drops any debt where `debt.startDate > cutoff.payDate`. The form defaults `startDate = today()` (`DebtFormViewModel.kt:21`). On 2026-05-12, current cutoff is the April-30 payday with windowEnd May 14 and payDate **April 30**. Any debt added "today" with the default start date has `startDate = May 12 > April 30 = payDate`, so it gets skipped. The user lands on Home, sees ₱2,250.50 due (the existing debts) but their newly-added ₱9,999,999.99 debt is invisible with no message. Repro: `snowball-form-long-pre-save.xml` → `snowball-home-three-debts.xml` — the third debt is in Debts but absent from Home payment list, while DUE stays at ₱2,250.50.
   - **Fix:** Either auto-set startDate to "first of the cutoff windowStart's month" when the user is importing a mid-stream debt, or render a one-line hint on Home ("3 debts start next cutoff — view in Debts") when there are debts with `startDate > payDate`. At minimum, the form's start-date helper text should say "this is the date you started paying — set to before your last payday to see it on Home this cutoff."

2. **Only one category exists and there is no way to add another.**
   `DatabaseFactory.kt:18-32` seeds "Credit Card" (SCHEDULED) and "MISC" (LEDGER). The form filters categories to SCHEDULED only (`DebtFormViewModel.kt:50`), so the dropdown effectively offers one option (verified in `snowball-form-cat-open.xml`: only "Credit Card" listed). `CategoryRepository.add()` exists but is never called from any screen (grep `categories\.add\(|addCategory` finds zero call-sites in `ui/`). Every debt is forced into "Credit Card" regardless of whether it is one. On Debts the category header `CREDIT CARD` becomes a noise band stamped above every row.
   - **Fix:** Either remove the category header on Debts when only one category is in use (treat single-category as flat-list), or add a category management entry point — most natural on Settings or as an inline "+ New category" item at the bottom of the dropdown.

3. **Validation failures on the form are silent.**
   `DebtFormViewModel.save()` returns `false` for blank name, missing category, non-numeric amount, due-day out of 1..31, etc. — but the only consequence in the UI is that `onSaved()` doesn't run, so the form sits there with no toast, no inline error, no border-color change, no shake. Repro: from `snowball-form-blank.xml`, tap Save → screen identical (`snowball-form-blank-after-save.xml`). A first-time user will tap Save, see nothing, and not know whether the app froze, whether their data was saved, or what is wrong.
   - **Fix (minimum):** disable the Save button until the required fields parse (name non-blank, amount > 0, total > 0, due in 1..31, category set). Better: surface inline `isError = true` on each `OutlinedTextField` and helper text under the offender. The `Save` button bg should also lose its Ice color when disabled — currently it stays `SnowColors.Ice` regardless of state.

4. **Delete is one-tap destructive with no confirmation and no undo.**
   `DebtFormScreen.kt:167-170` calls `vm.delete()` immediately on tap; no `AlertDialog`, no Snackbar with UNDO. In `snowball-after-delete.xml` Sloan and all its payment history vanished. The button is colored Ember (warning) but it lives alongside Cancel/Save in equal weight (each `Modifier.weight(1f)`), so it looks like any other action.
   - **Fix:** Wrap in an `AlertDialog` confirmation, or even better: tap-to-confirm pattern (the button text changes to "Tap again to delete" with a 3-second timeout — matches the swipe-to-confirm aesthetic). At minimum, show a Snackbar with UNDO action after deletion.

### Important (real polish gap)

5. **`LEFT OVER` reads the same color whether income covers debts or not.**
   `CutoffCard.kt:79` always passes `color = SnowColors.Ice` to the LEFT OVER cell. In `snowball-home-filled.xml` (income=0, due=2250.50), LEFT OVER shows ₱-2,250.50 in the same calm Ice as a positive surplus. A negative left-over should signal alarm — at minimum switch to `SnowColors.Ember` when `summary.breathingRoom < 0`, ideally also prefix with "Short by" instead of a minus sign.

6. **Form has no top bar / no Back affordance — Cancel is buried.**
   `DebtFormScreen.kt:48` is a flat `Column.verticalScroll`. The bottom-nav is hidden when route is `Route.Form` (`App.kt:83`), so there's no chrome at all. To dismiss without saving, a user must scroll all the way past Notes to reach Cancel. On Android 14 gesture nav, the system back-swipe from the edge collides with the swipe-right "undo" gesture on Home rows (verified — a left-edge swipe right on the Home payment row dismissed the app to launcher; `snowball-home-swipe-right-tried.xml` captured the launcher). Forms with no back button compound that.
   - **Fix:** Add a `TopAppBar` to the form with title "New debt" / "Edit debt" and a back arrow (left-aligned X or ChevronLeft). Move Save to the top-right (or keep at bottom but pin it). Cancel can disappear.

7. **Numeric inputs render `1500.0` when editing.**
   `DebtFormViewModel.kt:34` does `existing.monthlyAmount.toString()` — Double's toString adds `.0`. In `snowball-form-edit-sloan.xml`, MONTHLY AMOUNT shows `1500.0`. Aesthetically off-brand for a typography-forward app. Use the same formatter as `PesoText` (drop the fractional part when zero, else show two decimals).

8. **Settings income field has no save state and pre-fills `0.0`.**
   `SettingsScreen.kt:30` pre-fills with `initial.incomePerCutoff.toString()` (`0.0` by default) and writes on every keystroke. There is no "Save" or visual ack. A first-time user looking for "where do I enter income" sees a field already containing `0.0` and may not realize it's editable. Same `0.0` toString cleanup applies. Consider showing the formatted peso amount with thousand separators when not focused, and switching to a raw editable string only on focus.

9. **The "Payments already made" field is repurposed by the swipe gesture.**
   `DebtFormViewModel.kt:36` re-reads `repos.payments.countForDebt(existing.id)` so the field shows lifetime payments count, not the original backfill value the user typed. In `snowball-form-edit-sloan.xml` it shows `1` purely because I tapped a row once. This is correct from a data-integrity standpoint (you don't want a stale value) but the helper text "Use this to import a debt mid-way. Backfills payment history" no longer describes what the user is seeing in edit mode. Reword to "Total payments recorded (edit to reconcile)" or split into a read-only counter + a separate write-only backfill field.

10. **Long debt names break the Debts row's right-aligned amount.**
    `DebtsScreen.kt:83-106` lays out `Column(weight 1f) { name; subtitle }` next to `PesoText` in a centered Row. When the name wraps to 3+ lines (`snowball-debts-pre-edit.xml`: bounds `[53,584][1027,883]`, name extends to y=800), the amount is anchored vertically-centered and its top edge (`y=705`) sits inside the wrapped name area. The subtitle "Day 7 · 60 months" then runs at y=800-841, below where the amount ends. Visually it produces an L-shape that looks accidental.
    - **Fix:** Either `verticalAlignment = Alignment.Top` and right-anchor the amount to the first text line, or apply `maxLines = 2` + ellipsis on the name.

11. **The empty-state copy on Home tells the user to do something they may not be able to do.**
    `HomeScreen.kt:240-244` reads "No payments due this cutoff yet.\nAdd debts from the Debts tab." If income is also 0, that's a more pressing first step — Settings is silent on this. Empty Home with empty income is the literal first-use state and the only call-to-action sends them to Debts; income setup is undiscoverable from Home.
    - **Fix:** Make the empty hint a two-step onboarding: "1. Set your income in Settings. 2. Add debts in Debts." Or render an explicit CTA card linking to each.

12. **`Save` button keeps Ice background even when validation will fail.**
    See blocker (3). The button being visually identical in success/fail states is part of why the silent-failure is jarring. Even with a confirmation dialog or inline errors, the button should switch to a muted Frost-on-NightElev when disabled.

### Nitpicks

13. **PAYMENTS section header spacing inconsistency.**
    `HomeScreen.kt:61` puts `Spacer(24.dp)` above the section header but only `Spacer(2.dp)` between the "PAYMENTS" label and its italic instruction line, then `Spacer(8.dp)` before rows. The cutoff card uses 8dp / 24dp internally. The 2dp gap is too tight — at labelSmall 10sp tracking 4sp followed by bodySmall 12sp italic, the two lines visually collide. Bump to 6-8dp.

14. **`+` glyph as FAB is a typographic Plus, not an Add icon.**
    `DebtsScreen.kt:136` renders the FAB content as `Text("+", style = headlineLarge)`. The text-rendered `+` doesn't sit visually centered in the circle (Fraunces serifs at headlineLarge 28sp W500 inherit baseline offset, so the glyph drifts upward). Use `Icons.Outlined.Add` for vertical centering and TalkBack support.

15. **No contentDescription on the FAB, ProgressArc, or PesoText.**
    Grep finds 5 `contentDescription` usages total (`ui/nav/Nav.kt:60`, two `null` in DebtsScreen/DebtFormScreen icon decorations, two on Home action-background icons). The FAB has no semantics — TalkBack will announce "plus" with no context. ProgressArc has no role/state semantics so a screen reader user can't tell whether a row is paid. Add `Modifier.semantics { contentDescription = "Add debt" }` to the FAB; add `role = Role.Checkbox` + `stateDescription` to each payment row.

16. **"Use last day of month" switch is unlabeled for accessibility.**
    The Switch is `NAF="true"` in `snowball-form-blank.xml` (bounds `[53,1585][190,1711]`). The label `Use last day of month (Feb adjusts)` is a sibling Text, not part of the Switch semantics. TalkBack will say "Switch, off" with no description.

17. **Bottom-nav active state is one shade of Frost vs FrostDim.**
    `Nav.kt:50-69` distinguishes selected/unselected only by `Frost (#F2F5F8)` vs `FrostDim (#5E6874)` tint. There is no indicator pill, no underline, no bg color change. On a screen at arm's length the active tab can read identical to inactive. Add a 2dp top accent or a subtle background tint for the selected tab.

18. **"View archived" tap region has invisible affordance.**
    `DebtsScreen.kt:54-59` is a bare `Text(...)` with `.clickable` — no underline, no chevron, no background. The Ice color is the only signal. Users won't know it's tappable until they accidentally tap or notice the heading change.

19. **Form labels are ALL-CAPS labelSmall with 3sp letter-spacing — verging on hard to read.**
    `DebtFormScreen.kt:189-191` uppercase + 3sp tracking is the right call for the cutoff card section labels, but for form field labels (NAME, CATEGORY, MONTHLY AMOUNT...) it bleeds into "wedding invitation" territory. Material-style sentence-case labels above `OutlinedTextField` would feel less precious and parse faster.

20. **The cutoff card "DUE" label sits 8dp above a 96sp number with no breath beneath.**
    `CutoffCard.kt:63-70`: `Spacer(8dp)` between "DUE" label and the giant PesoText, but `Spacer(24dp)` between the PesoText and the LEDGER row. That 8dp top / 24dp bottom is asymmetric — the DUE numeral feels visually shoved up against its label. Try 12dp top / 20dp bottom.

21. **The italic "Swipe left to mark paid · swipe right to undo" copy is partially decorative.**
    Once a user has done it twice, they'll stop reading it. As of v0.1 it lives on prime real estate at 12sp italic FrostDeep (1.65:1 contrast — see Accessibility). Move to a coachmark on first launch and dismiss thereafter.

22. **Debts list does not show debt progress.**
    Each card shows monthly amount and "Day 10 · 12 months" — but not "3 of 12 paid". The payment count is already in the data (form re-derives it). Adding `n / total` or a thin progress bar at card bottom would be a substantial improvement at low cost.

## Accessibility findings
- **Color contrast (computed from `SnowColors`):**
  - `FrostDeep #3C4452` on `Night #0B0F14`: ~1.65:1 — **fails** WCAG AA for all text. Used by:
    - "Swipe left to mark paid · swipe right to undo" caption on Home (`HomeScreen.kt:71`).
    - Form placeholder "0" in Payments-already-made (`DebtFormScreen.kt:111`).
    - "Snowball v0.1" version stamp on Settings (`SettingsScreen.kt:70`).
  - `FrostDim #5E6874` on `Night`: ~2.7:1 — **fails AA** for normal body, **passes AA-large** (3:1) for the displayLarge ₱ and headlineLarge category labels but is below 4.5:1 for the labelSmall section caps. Used by section labels (PAYMENTS, INCOME, DUE), inactive amount color, dropdown chevron.
  - `Frost #F2F5F8` on `Night`: ~19:1 ✓.
  - `Ice #9FCEE3` on `Night`: ~12.5:1 ✓.
  - `FrostMute #A8B2BF` on `Night`: ~9:1 ✓.
  - The 1dp `Line #12FFFFFF` (Alpha 12) divider between payment rows is effectively invisible (about 4% white over Night).
- **NAF (Not Accessibility Friendly) flags in uiautomator:**
  - Every `OutlinedTextField` in the form (Name, Category, Monthly amount, Total payments, Due day, Switch, Notes) is `NAF=true`. The visible label is a separate `Text` composable, so semantics aren't linked. TalkBack will read the field as "edit box" with no name.
  - **Fix:** Pass `label = { Text(...) }` to `OutlinedTextField` (Material 3's built-in floating-label slot wires semantics automatically) and remove the separate `Field` wrapper, or wrap the field in `Modifier.semantics { contentDescription = "Name" }`.
- **Touch targets (density 420, 1dp = 2.625px, 48dp = 126px):**
  - FAB 56×56dp ✓.
  - Bottom-nav each 137×48dp ✓.
  - "View archived" 82×48dp ✓ (just barely — driven by the parent Row's vertical padding).
  - Payment rows ~70dp tall ✓.
  - Form fields 56dp tall ✓.
- **Text scaling at `font_scale=1.30`:**
  - In `snowball-home-font130.xml`, the cutoff card's display-large ₱ glyph grows from `[116,643][214,814]` (1.0) to `[116,953][214,1156]` — its bottom edge (1156) is below the LEDGER row top (1240) but the card height pushed the LEDGER row down by ~150 px. Looks tight but no clipping visible.
  - The PAYMENTS section header descends to y=1667 vs y=1266 at 1.0. The Sloan row's bottom edge at 1.30 reaches y=2175, exactly where the bottom-nav starts at y=2175. **The last payment row touches the nav with zero margin.** Verify with screenshot. A third row at font_scale 1.30 would be partially obscured. Add `Modifier.padding(bottom = 16.dp)` to the scroll column or ensure `imePadding` / nav-bar inset.
- **Rotation:** AndroidManifest allows rotation (`android:configChanges="orientation|screenSize|keyboardHidden"`) but the emulator stayed at rotation=0 under `user_rotation 1`. Cannot verify landscape layout. The `Column.verticalScroll` design will scale (everything's percentage-based) but the cutoff card may look short and wide.

## Suggested priority order
1. **Stop silently dropping new debts from Home.** Either change form default `startDate` semantics or add a visible "debts not yet active" line.
2. **Wire validation feedback into the form.** Disable Save, color the button, add per-field error states.
3. **Confirm deletes.** AlertDialog or two-tap confirmation. The current behavior is data-loss-prone.
4. **Make negative LEFT OVER look negative.** Switch to Ember and a "Short by" prefix when `breathingRoom < 0`.
5. **Add a Back affordance to the form.** A simple `TopAppBar` with title + back arrow solves "Cancel is buried" and "no chrome" simultaneously. While there, give the form a title that names what's being edited.

## Inspection methodology
This review was conducted **without visually reading any screenshot** — the inspector's tools do not include image rendering, and the project's prior subagent crashed on a PNG Read, so I treated screenshots as artifacts for the human reviewer only.

My visual judgments come from three sources:
1. **Compose source code** under `composeApp/src/commonMain/kotlin/com/snowball/ui/` — I read `App.kt`, `Nav.kt`, `HomeScreen.kt`, `DebtsScreen.kt`, `DebtFormScreen.kt`, `SettingsScreen.kt`, the four ViewModels, `components/CutoffCard.kt`, `PesoText.kt`, `HeroAmount.kt`, `ProgressArc.kt`, `CategoryIcon.kt`, and the theme files `Color.kt`, `Type.kt`, `Theme.kt`. Dimensions (dp, sp), exact color hex values, padding, spacing, weights, and validation logic were pulled from those files.
2. **Domain layer** — `Cutoff.kt`, `CutoffCalculator.kt`, `DateUtils.kt`, `data/db/DatabaseFactory.kt`, `data/repo/CategoryRepository.kt` to verify why a debt drops off Home and to confirm seeded categories.
3. **uiautomator XML dumps** for every meaningful state, captured immediately after each navigation, providing exact `bounds=[x,y][x,y]` for every visible node. These were my "visual proxy" — I used them to verify which elements appeared on screen, their pixel-precise layout, touch-target sizes, and overlap with system chrome. The XML files listed in the artifact table above are the ones I read.

Flows exercised via `adb shell input tap / input text / input swipe`:
- Cleared install → Home empty → bottom-nav to Debts empty → FAB → form blank → Save (silent fail) → typed Name "Sloan", selected Credit Card, ₱1500, 12 months, due 10, start 2026-04-10 → Save → Debts (1 debt) → FAB → second debt → Home (2 rows showing) → swipe-left on row → tap-toggle on row → bottom-nav to Settings → typed income 25000 → back to Home (LEFT OVER updated) → bottom-nav to Debts → FAB → third debt with default startDate (today) → Save → Home (third debt missing) → bottom-nav to Debts → tap Sloan card → Edit form (Payments already made = 1 confirms swipe persisted) → scroll → Delete (no confirmation, instant) → confirmed Sloan gone in Debts list.
- Side flows: `pm clear` start; `font_scale 1.30` relaunch; `user_rotation 1` attempt (emulator did not actually rotate); blank-form Save attempt; left-edge swipe-right test (collided with system back gesture, dismissed app).

I did **not** read `docs/superpowers/specs/`, `docs/superpowers/plans/`, `docs/design/` existing files, `.superpowers/`, or `HANDOFF.md`, per the unbiased-evaluation directive. I did not read any `.png` file at any point.
